# Phase 03.1: Gerar confrontos das oitavas por cupom a partir dos palpites da fase de grupos - Research

**Researched:** 2026-03-29
**Domain:** Derivacao de bracket por cupom, API autenticada para confrontos previstos e integracao do mata-mata na CupomView
**Confidence:** HIGH

## Summary

O gap atual nao e mais de desbloqueio de fase. O backend ja impede eliminatorias antes do preenchimento completo dos grupos, mas ainda opera com `Jogo` estatico para definir participantes e classificado. Isso quebra o caso real do cupom: mesmo com todos os 72 palpites de grupos preenchidos, as oitavas continuam sem confrontos corretos por cupom.

O seed do torneio deixa isso explicito. Existem apenas jogos eliminatorios estaticos de exemplo em `TorneioMockadoSeeder.php`, com selecoes fixas como `BRA x JPN`, e a `CupomView` renderiza mata-mata a partir de `torneio.jogos`. Esse desenho serve para mock visual, mas nao para o fluxo funcional exigido agora.

**Recomendacao principal:** dividir a fase em duas ondas:
- onda 1: criar um servico backend dedicado para derivar a tabela dos grupos e montar o bracket do cupom, expor isso por API autenticada e ajustar a persistencia do mata-mata para resolver o classificado a partir dos participantes derivados;
- onda 2: adaptar a `CupomView` e os tipos TypeScript para consumir o bracket do cupom, renderizar as equipes derivadas e manter resumo/penalidades/desbloqueio progressivo sobre essa fonte de verdade.

## Existing Architecture Findings

### Backend atual
- `ServicoApostas` resolve `selecao_classificada_id` lendo `Jogo::selecao_mandante_id` e `Jogo::selecao_visitante_id`. Isso falha quando os participantes devem vir do bracket previsto do cupom.
- `ServicoFechamentoApostas` ja implementa a progressao grupos -> oitavas -> quartas -> semis -> finais por contagem de apostas. O que falta e o bracket real.
- `CupomController@show` hoje devolve apenas `cupom`, sem dados derivados de bracket.
- `ApostaController@index` devolve apenas apostas persistidas; a `CupomView` combina isso com `/torneio`.

### Frontend atual
- `CupomView.vue` calcula `fasesRodadas`, `jogosFaseAtual`, `resumoBracket` e `autoSalvar` com base em `torneio.value.jogos`.
- A tipagem em `frontend/src/tipos.ts` nao possui nenhum tipo para participantes previstos por cupom nem estrutura de bracket derivado.
- O resumo de campeao/vice/terceiro usa `final` e `terceiro_lugar` do torneio publicado, logo ainda assume confrontos estaticos.

### Seed / dados mockados
- `TorneioMockadoSeeder.php` cria 72 jogos de grupos, mas apenas 1 jogo para oitavas, 1 para quartas, 1 semifinal, 1 terceiro lugar e 1 final.
- Isso confirma que a camada de bracket do cupom nao pode depender do seed estatico para representar todo o mata-mata real.

## Recommended Technical Direction

### 1. Novo servico backend para bracket do cupom
Criar um servico dedicado, por exemplo `ServicoBracketCupom`, responsavel por:
- carregar apostas de grupos e eliminatorias do cupom;
- calcular classificacao de cada grupo a partir dos placares apostados;
- mapear 1o e 2o colocados para uma tabela oficial de oitavas;
- derivar quartas, semifinais, terceiro lugar e final a partir dos classificados previstos nas apostas eliminatorias do proprio cupom;
- devolver uma estrutura uniforme de partidas previstas, com `jogo_base_id`, `fase_slug`, `ordem_na_fase`, `selecao_mandante_prevista`, `selecao_visitante_prevista`, `bloqueado`, `motivo_bloqueio`.

### 2. API autenticada separada do torneio publico
O torneio publico pode continuar estatico. A camada prevista por cupom deve entrar em rota autenticada, por exemplo:
- `GET /cupons/{cupom}/bracket`

Essa rota evita misturar dados publicos do torneio com dados privados/derivados do cupom e simplifica a `CupomView`.

### 3. Persistencia do mata-mata precisa consultar o bracket previsto
`ServicoApostas` nao pode mais derivar classificado de jogos eliminatorios olhando apenas para `Jogo`. O caminho correto e:
- localizar o jogo base;
- pedir ao servico de bracket os participantes previstos daquele jogo para o cupom;
- usar os ids previstos para derivar `selecao_classificada_id`.

### 4. CupomView deve separar "jogo base" de "confronto previsto"
No frontend, a fase eliminatoria deve ser renderizada a partir da resposta do bracket do cupom, nao de `torneio.jogos`. Os dados publicos do torneio seguem uteis para:
- grupos, fases, rodadas, regras;
- horarios e metadata do jogo base;
- ranking e resultado real.

## Anti-Patterns to Avoid

- Reaproveitar `torneio.jogos` como fonte dos participantes eliminatorios previstos.
- Colocar o calculo da classificacao dos grupos no frontend.
- Persistir selecoes previstas sobrescrevendo `jogos` base do torneio.
- Acoplar o bracket do cupom a dados fixos do seed, como `BRA x JPN`.

## Common Pitfalls

### Pitfall 1: derivacao parcial das oitavas
**What goes wrong:** o sistema destrava a fase mas monta confrontos com `null` ou equipes erradas porque alguns grupos nao foram calculados corretamente.
**How to avoid:** o servico deve validar que todos os grupos da fase de grupos do torneio possuem exatamente dois classificados derivados antes de montar oitavas.

### Pitfall 2: quartas e fases seguintes continuarem lendo o jogo base
**What goes wrong:** oitavas aparecem corretas, mas quartas/semis/final continuam exibindo selecoes estaticas do seed.
**How to avoid:** usar a mesma estrutura de bracket derivado para todas as fases eliminatorias, nao apenas para oitavas.

### Pitfall 3: classificado salvo com ids do seed estatico
**What goes wrong:** o usuario palpita `Mexico x Suica` no confronto previsto, mas `ServicoApostas` salva o classificado de `BRA x JPN`.
**How to avoid:** derivar `selecao_classificada_id` sempre a partir dos participantes previstos retornados pelo servico do cupom.

### Pitfall 4: inconsistencias entre API de leitura e autosave
**What goes wrong:** a UI renderiza o bracket derivado por uma rota, mas o autosave e o resumo ainda leem outra fonte de dados.
**How to avoid:** centralizar no frontend um conjunto `jogosDoCupom`, usando grupos do torneio e eliminatorias do bracket derivado.
